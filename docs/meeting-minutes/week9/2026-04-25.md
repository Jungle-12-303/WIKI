## 팀별 공유 내용

- 홍윤기: 협업은 기능별로 세분화해서 나누는 방식이 적절하다고 봄.
- 정찬빈: 알람클럭을 구현 중이며 계획대로 진행 중. 서로 모르는 점을 알려주면서 작업하고 있고, 실력 차이는 크지 않다고 느낌. 현재는 에러가 있어 merge보다 에러 해결을 우선할 예정. 디버깅과 트래킹을 적극적으로 보는 편.
- 정범진: 실력 차이는 아주 크지 않지만 전체적으로 어려움을 느끼고 있고, 다른 팀의 도움도 받았음. 4명이 한 컴퓨터에서 같이 작업하는 방식으로 진행 중. 다만 시간이 오래 걸리고, 도와줄 사람이 없으면 진행이 느려지는 단점이 있음.
- 이창원: 현재 이슈 단위로 코드를 나누고 있으나, 지금 방식이 맞는지 다시 검토가 필요하다고 봄. 팀원 간 편차가 있어 분업 방식을 다시 고민 중.
- 임가인: 테스트 중심으로 진행 중. 먼저 전체 흐름을 훑고 테스트 통과 기준으로 구현하고 있음. Git이 아직 익숙하지 않아 `main` 중심으로 작업하고 백업본을 따로 두는 방식으로 운영 중. 월요일까지 구현 후 코어타임에 함께 코드를 볼 계획.
- 임재환: 알람클럭을 함께 구현 중이며 현재 팀 전체가 막혀 있는 상태. AI로 pseudo code와 흐름도를 만들어 전체 사이클을 먼저 잡아보는 중. 팀원 브랜치를 `dev`에 통합하고, 테스트가 통과하면 `main`에 merge하는 전략을 사용 중.
- 고유진: pair programming을 계획했지만 둘 다 길을 잃으면 같이 막힌다는 한계를 느낌. 학습 단계에서는 AI를 쓰지 않으려는 입장. 코어타임에서는 구조적인 이야기와 설계 방향에 대한 대화를 많이 했고, 객체지향적 구조에 대한 요구도 있었음.

## 협업 방식 정리

- 현재 협업 방식은 크게 기능/이슈 단위 분업, 페어 프로그래밍, 전원 공동 작업, 테스트 중심 진행으로 나뉨.
- 실력과 과제 이해도가 비슷한 팀은 분업을 시도하고, 편차가 크거나 전체 흐름이 약한 팀은 함께 구현하는 쪽이 더 현실적이라는 의견이 많았음.
- 분업은 서로의 실력을 믿을 수 있을 때 잘 작동하고, 그렇지 않으면 한 번 더 같이 이해를 맞추는 과정이 필요하다는 인식이 공유됨.

## 실력차 극복에 대한 논의

- 개념을 먼저 이해하는 것이 오래 걸리므로, 빠른 사람이 먼저 개념과 설계를 정리한 뒤 전체 흐름을 설명해주는 방식이 효과적이라는 의견이 있었음.
- 코드 레벨을 처음부터 깊게 보기보다, 전체 흐름과 OS 개념을 먼저 잡고 막힐 때 세부 구현을 보는 방식이 더 낫다는 공감대가 있었음.
- 잘하는 인원이 다른 팀을 잠깐 도와주는 방식도 제안됨. 예시로 상대적으로 여유 있는 인원 한 명이 범진 팀을 지원하는 방안이 언급됨.
- 발표나 설명 역할이 특정인에게만 희생이 되지 않도록, 왜 코어타임을 하는지와 발표자가 얻는 점을 함께 정리할 필요가 있음.
- `강의 + 개인 강의` 형태로 개념을 다시 설명하면 반 전체 이해력이 올라갈 수 있다는 의견이 있었음.

## 공통 메모

- Pintos는 단순 코딩 과제가 아니라 개념 이해, 설계, 코드 작성 전부를 연습하는 과제라는 인식이 있었음.
- 학습 초반에는 구현에 필요한 `struct`나 자료구조를 미리 알면 도움이 된다는 의견이 있었음.
- `gdb`를 VS Code와 연결해 브레이크포인트를 잡는 방식이 유용하다는 의견이 있었음.
- 테스트는 `compass`에 있는 항목을 적극 활용하면 편하다는 의견이 있었음.
- 앞으로 무엇을 할지, 다음 단계가 무엇인지가 팀원들에게 더 잘 보이도록 공유가 필요하다는 의견이 있었음.
