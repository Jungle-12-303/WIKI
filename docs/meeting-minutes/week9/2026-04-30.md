| 안건 | 결정 상태 | 결정 내용 |
|------|-----------|-----------|
| 각 팀 진행 상황 및 다음 주 진행 방식 | 논의 중 | 팀별로 현재 진행 방식과 문제점을 공유했다. 공통적으로 진도 차이, 소통 방식, 코드 컨벤션, 커밋/리뷰 방식 개선이 필요하다는 의견이 나왔다. |
| 1팀 진행 방식 | 결정 | 하루 단위 목표를 정하고, 10시 전 개인 브랜치 커밋 후 코드 리뷰 및 머지를 진행한다. 진도가 빠른 사람의 코드는 느린 사람이 보고 이해하는 방향으로 맞춘다. |
| 2팀 진행 방식 | 결정 | `argument pass`는 함께 진행하고, 이후 작업은 나누어 구현한다. 모르는 부분은 기존처럼 호출하거나 짝을 지어 해결한다. |
| 3팀 진행 방식 | 결정 | `priority-sema`까지 구현한다. 프로젝트를 새로 받아 처음부터 진행한 뒤 9주차와 10주차 내용을 합친다. 코드 컨벤션을 엄격히 지키고, 커밋을 자주 하며 PR merge 보호 규칙을 설정한다. 2시간 간격으로 진행 상황을 점검한다. |
| 4팀 진행 방식 | 결정 | 자리 배치를 변경하고 페어 프로그래밍 멤버를 바꾼다. 둘씩 역할을 나누고 페어끼리 진도를 맞춘다. 코드 컨벤션은 PintOS 기준으로 맞춘다. |
| 5팀 진행 방식 | 논의 중 | 구체적인 계획은 없으며, 이번 주도 비슷하게 진행한다. 먼저 진행한 팀원이 다른 팀원의 진도를 끌어올리는 방식으로 진행한다. |
| 6팀 진행 방식 | 보류 | 아직 회고를 하지 못했으며, PintOS 특강 이후 진행 방식을 정하기로 했다. |
| 7팀 진행 방식 | 결정 | 강의 후 GitBook 기반으로 테스트 케이스를 확인하고 개념을 정리한다. 2시간 간격으로 30분 코어타임을 운영하며, 토요일과 월요일은 페어 프로그래밍, 화요일과 수요일은 개인 진행으로 한다. |
| 발표 내용 구성 | 결정 | 협업, 코드 구현, 개념 설명의 균형을 맞춘다. 발표 앞뒤에 협업 및 회고 내용을 짧게 포함하고, 어려웠던 점이나 착각했던 점 등 트러블슈팅 중심 내용을 넣는다. 마지막 장에는 Q&A 페이지를 넣는다. |
| 발표 디자인 | 결정 | 발표 디자인은 각 팀이 자유롭게 정한다. |
| WIKI 작성 방식 | 결정 | 테스트 케이스별로 정리하고, 삽질한 내용이나 팁을 짧게 작성한다. 폴더를 만든 뒤 팀장이 먼저 작성해 참여를 유도한다. |
| 추가 의견 | 보류 | 세션 관련 내용은 일단 보류한다. |
| 코치님 팁 공유 | 결정 | 지난주 내용이 구현되지 않아도 진행은 가능하지만 `wait`가 오래 걸려 느릴 수 있다. 잘 구현된 코드를 참고해서 진행하는 것이 좋다. |
