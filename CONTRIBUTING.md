# WIKI 기여 가이드

이 문서는 WIKI 레포에 학습 내용을 작성할 때 따라야 할 모든 규칙을 정리합니다.
처음 기여하는 사람도 이 문서만 읽으면 바로 작업을 시작할 수 있도록 작성되었습니다.

---

## 1. 폴더 구조

```
docs/
  cs-기초/              # L0: 카테고리
    README.md           # 카테고리 목차 (키워드 목록 + 상태)
    32bit-64bit.md      # L1: 키워드 파일
    memory-structure.md
    ...
  algorithm/
  malloc-lab/
  네트워크/
  os/
  pintos/
    project0/           # L1: 하위 폴더 (프로젝트별 분류)
      README.md         # L1 목차
      virtual-machine.md # L2: 키워드 파일
      ...
    project1-threads/
    project2-userprog/
    project3-vm/
    project4-filesys/
  ai/
  db/
  web-security/
  팀운영/
    컨벤션/
    외부리소스/
  meeting-minutes/
    week9/
      2026-04-25.md
```

최대 깊이는 L0부터 L9까지 (10레벨) 허용합니다.
현재 대부분의 키워드는 L0 > L1 (카테고리 > 키워드 파일) 구조이고, `pintos/`처럼 프로젝트 단위로 한 단계 더 들어가는 경우가 있습니다.

## 2. 파일 네이밍 규칙

**파일명:**

- 영문 소문자와 하이픈(-)만 사용합니다.
- 공백, 대문자, 언더스코어(_)는 사용하지 않습니다.
- 확장자는 `.md`입니다.

| 올바른 파일명 | 잘못된 파일명 | 이유 |
|---------------|---------------|------|
| `cpu-vs-gpu.md` | `CPU_vs_GPU.md` | 대문자, 언더스코어 사용 |
| `hash-table.md` | `hash table.md` | 공백 사용 |
| `deep-shallow-copy.md` | `deepShallowCopy.md` | 카멜케이스 사용 |
| `context-switching.md` | `context_switching.md` | 언더스코어 사용 |

**폴더명:**

- 영문 소문자 + 하이픈을 기본으로 합니다.
- 카테고리 폴더명에 한글은 허용합니다 (`cs-기초/`, `네트워크/`, `팀운영/`).
- 모든 폴더에는 반드시 `README.md`가 있어야 합니다.

## 3. 키워드 파일 템플릿

새 키워드 파일을 만들 때는 아래 템플릿을 복사해서 사용하세요.

```markdown
# 키워드 제목

> 한 줄 요약 설명 (이 키워드가 무엇인지 한 문장으로)

## 개요

이 키워드가 왜 중요한지, 어떤 맥락에서 등장하는지 2~3문장으로 설명합니다.

## 상세 설명

본문 내용을 작성합니다.
필요에 따라 소제목(###)을 추가해서 구분하세요.

## 참고 자료

- [링크 제목](URL)
```

**선택적으로 추가할 수 있는 섹션:**

| 섹션 | 언제 추가하나 | 예시 |
|------|---------------|------|
| `## 코드 예시` | 코드로 설명하는 것이 더 명확할 때 | C 코드로 포인터 사용법 시연 |
| `## 비교 표` | 두 개념을 나란히 비교할 때 | Stack vs Heap, DFS vs BFS |
| `## 생각해볼 질문` | 학습 후 스스로 점검할 질문 | "왜 Round Robin은 time quantum이 필요한가?" |
| `## 관련 키워드` | 이 문서와 연결되는 다른 키워드가 있을 때 | `[프로세스와 스레드](../os/process-thread.md)` |

**작성 시 주의사항:**

- `_작성 필요_`를 그대로 남기지 마세요. 내용을 채우거나 섹션 자체를 삭제하세요.
- 외부 이미지 링크는 끊어질 수 있으니, 가능하면 텍스트와 코드로 설명하세요.
- 다른 사람이 읽는 문서입니다. 본인만 알 수 있는 약어나 메모 형식은 피하세요.

## 4. README.md (목차) 템플릿

각 폴더의 `README.md`는 해당 폴더에 어떤 키워드가 있고, 각각의 작성 상태가 어떤지를 보여주는 목차입니다.

```markdown
# 카테고리 이름

카테고리에 대한 한 줄 설명.

| 키워드 | 파일 | 상태 |
|--------|------|------|
| 32bit vs 64bit | [32bit-64bit.md](32bit-64bit.md) | 완료 |
| 메모리 구조 | [memory-structure.md](memory-structure.md) | 작성 중 (@홍윤기) |
| 가비지 컬렉션 | [garbage-collection.md](garbage-collection.md) | 작성 필요 |
```

**규칙:**

- 표의 순서는 파일 생성 순서를 따릅니다 (나중에 추가된 키워드가 아래에 옵니다).
- 링크는 상대경로를 사용합니다 (`[파일명.md](파일명.md)`).
- 하위 폴더가 있다면 표 아래에 폴더 링크를 별도로 표기합니다.

---

## 5. 작성 상태 관리

키워드 작성 상태는 세 단계로 나뉩니다.

| 상태 | 의미 | 표기 예시 |
|------|------|-----------|
| `작성 필요` | 파일은 있지만 내용이 비어 있음 | `작성 필요` |
| `작성 중 (@이름)` | 누군가 현재 작성 중 | `작성 중 (@정찬빈)` |
| `완료` | 내용 작성이 끝남 | `완료` |

**상태 변경 타이밍:**

1. 키워드를 고르면 → 즉시 해당 README.md에서 상태를 `작성 중 (@본인이름)`으로 변경
2. 상태 변경만 먼저 커밋하고 push
3. 키워드 내용을 작성
4. 작성이 끝나면 상태를 `완료`로 변경하고 커밋

상태를 먼저 변경하는 이유: 다른 사람이 같은 키워드를 중복으로 고르는 것을 막기 위해서입니다.

## 6. 충돌 방지 규칙

여러 사람이 동시에 작업하기 때문에 git 충돌을 최소화하는 규칙이 필요합니다.

### 규칙 1: 하나의 키워드 파일은 한 사람만 작성

- `작성 중 (@이름)` 상태인 파일은 다른 사람이 수정하지 않습니다.
- 수정이 필요하면 담당자에게 직접 이야기하세요.

### 규칙 2: README.md 수정은 최소한으로

README.md(목차)는 충돌이 가장 많이 발생하는 파일입니다. 다음 경우에만 수정하세요:

- 키워드 상태 변경 (`작성 필요` → `작성 중` → `완료`)
- 새 키워드 행 추가

내용 작성은 반드시 개별 키워드 파일에서 합니다.

### 규칙 3: 커밋 단위를 잘 묶기

| 상황 | 하나의 커밋에 포함할 파일 |
|------|---------------------------|
| 키워드 작성 시작 | `README.md` (상태 변경) |
| 키워드 내용 작성 완료 | `키워드.md` + `README.md` (상태를 완료로) |
| 새 키워드 추가 | `키워드.md` (새 파일) + `README.md` (행 추가) |
| 하위 폴더 추가 | `새폴더/README.md` + `새폴더/키워드.md` + `상위 README.md` (링크 추가) |

### 규칙 4: push 전에 pull

```bash
git pull --rebase origin main
```

커밋 후 push 전에 반드시 원격의 최신 상태를 가져옵니다. 충돌이 발생하면 해결 후 push하세요.

### 충돌이 발생했을 때

README.md 표에서 충돌이 발생한 경우, 양쪽 행을 모두 살리면 대부분 해결됩니다.
키워드 파일 자체에서 충돌이 발생했다면 담당자끼리 직접 논의하세요.

## 7. 커밋 메시지 규칙

이 레포는 [Conventional Commits](https://www.conventionalcommits.org/) 형식을 사용하되, 제목과 본문을 한국어로 작성합니다.

### 형식

```
<type>: <한국어 제목>
```

본문이 필요한 경우:

```
<type>: <한국어 제목>

<한국어 본문>
```

### scope는 사용하지 않습니다

`docs(os):` 처럼 scope를 쓰지 않습니다. 변경된 파일 경로와 한국어 제목으로 영역을 충분히 전달합니다.

### 타입

| 타입 | 용도 | 이 레포에서의 예시 |
|------|------|--------------------|
| `docs` | 키워드 작성, 문서 변경 | 키워드 내용 작성, 회의록 추가 |
| `chore` | 폴더 구조 변경, 설정 파일 | 새 카테고리 폴더 생성, .gitignore 수정 |
| `feat` | 새로운 스크립트나 도구 | lint 스크립트 추가 |
| `fix` | 오류 수정 | 깨진 링크 수정, 오타 수정 |
| `refactor` | 구조 개선 | 키워드 파일을 하위 폴더로 분리 |
| `style` | 포맷팅 | 표 정렬, 줄바꿈 통일 |

### 제목 작성 규칙

- 한 줄로 씁니다.
- 마침표를 찍지 않습니다.
- 결과를 설명합니다 (작업 행위가 아니라).
- `수정`, `업데이트`, `작업`, `변경` 같은 막연한 단어를 피합니다.

**좋은 예시:**

```
docs: CPU vs GPU 키워드를 작성
docs: week10 팀장 회의록을 추가
chore: pintos/project2 폴더 구조를 생성
fix: os/README.md의 깨진 링크를 수정
refactor: scheduler 키워드를 하위 폴더로 분리
```

**나쁜 예시:**

```
docs: 수정                    # 무엇을 수정했는지 알 수 없음
docs: 업데이트                 # 무엇이 업데이트되었는지 알 수 없음
docs: 작업 중                  # 결과가 아닌 상태를 설명
Update README.md              # 영어 제목, 막연한 내용
```

### 본문

본문은 선택입니다. 다음 경우에 추가하세요:

- 변경 이유가 제목만으로 전달되지 않을 때
- 여러 파일을 한 커밋에 묶었을 때 맥락 설명이 필요할 때

본문도 한국어로 작성합니다. 제목과 본문 사이에 빈 줄을 넣습니다.

## 8. 브랜치 전략

- 기본 작업은 `main` 브랜치에서 직접 합니다. 이 레포는 학습 문서 레포이므로 PR 없이 직접 push하는 것을 허용합니다.
- 대규모 변경 (카테고리 전체 재구성, 폴더 구조 변경 등)은 별도 브랜치를 만들어 PR을 올려주세요.

**대규모 변경의 기준:** 3개 이상의 폴더에 걸친 구조 변경, README.md 5개 이상 동시 수정, 파일 이름 일괄 변경 등.

```bash
# 대규모 변경 시
git checkout -b refactor/reorganize-pintos
# 작업 후
git push origin refactor/reorganize-pintos
# GitHub에서 PR 생성
```

## 9. 하위 레벨 추가 방법

키워드 하나가 너무 커져서 여러 파일로 나눠야 할 때, 아래 절차를 따릅니다.

### 절차

**예시: `os/` 폴더의 `scheduler.md`를 하위 폴더로 분리**

1. 기존 파일 내용을 확인합니다.

2. 하위 폴더를 만들고 README.md와 키워드 파일을 생성합니다:

```bash
mkdir docs/os/scheduler
```

3. `docs/os/scheduler/README.md`를 작성합니다:

```markdown
# CPU 스케줄링 상세

CPU 스케줄링 알고리즘별 상세 설명.

| 키워드 | 파일 | 상태 |
|--------|------|------|
| Round Robin | [round-robin.md](round-robin.md) | 작성 필요 |
| Priority Scheduling | [priority.md](priority.md) | 작성 필요 |
| MLFQ | [mlfq.md](mlfq.md) | 작성 필요 |
```

4. 하위 키워드 파일을 생성합니다 (키워드 파일 템플릿 사용).

5. 상위 `docs/os/README.md`에서 기존 행을 폴더 링크로 변경합니다:

```markdown
| CPU 스케줄링 | [scheduler/](scheduler/) | 하위 키워드 참조 |
```

6. 기존 `scheduler.md`의 내용을 적절한 하위 파일로 옮깁니다.

7. 이 모든 변경을 하나의 커밋으로 묶습니다:

```bash
git add docs/os/scheduler/ docs/os/README.md
git rm docs/os/scheduler.md  # 기존 파일 삭제 (내용을 옮겼으므로)
git commit -m "refactor: os/scheduler 키워드를 하위 폴더로 분리"
```

## 10. 회의록 규칙

### 위치

```
docs/meeting-minutes/
  week9/
    2026-04-25.md
  week10/
    2026-05-02.md
```

### 파일명

`YYYY-MM-DD.md` 형식입니다. 주차 폴더(`weekN/`) 안에 넣습니다.

### 작성 형식

```markdown
# 팀장 회의록 -- YYYY-MM-DD (요일)

**참석자:** 이름1, 이름2, ...

**주제:** 논의 주제 요약

| 안건 | 결정 상태 | 결정 내용 |
|------|-----------|-----------|
| 안건 1 | 결정 / 논의 중 / 보류 | 결정 또는 논의 내용 요약 |

## 팀별 공유 내용

- 이름: 공유 내용 요약

## 할 일

- [ ] 할 일 1
- [ ] 할 일 2

## 다음 회의에서 더 논의할 질문

- 질문 1
```

### 커밋 예시

```bash
git add docs/meeting-minutes/week10/2026-05-02.md
git commit -m "docs: week10 2026-05-02 팀장 회의록을 추가"
```

## 11. 새 카테고리 추가

기존 카테고리에 맞지 않는 주제가 생겼을 때:

1. `docs/` 아래에 새 폴더를 만듭니다.
2. 폴더 안에 `README.md`(목차)를 작성합니다.
3. `docs/README.md`(최상위 목차)에 새 카테고리 링크를 추가합니다.
4. 하나의 커밋으로 묶습니다.

```bash
mkdir docs/새카테고리
# README.md 작성
git add docs/새카테고리/README.md docs/README.md
git commit -m "chore: 새카테고리 폴더 구조를 생성"
```

새 카테고리를 추가하기 전에 기존 카테고리의 하위 레벨로 넣을 수 있는지 먼저 검토하세요.

## 12. 자주 하는 실수와 해결

| 실수 | 올바른 방법 |
|------|-------------|
| README.md 상태를 안 바꾸고 키워드 작성 시작 | 반드시 상태 먼저 변경 후 push |
| 키워드 내용을 README.md에 직접 작성 | README.md는 목차만, 내용은 키워드 파일에 |
| 커밋 메시지를 영어로 작성 | 타입만 영어, 제목은 한국어 |
| 파일명에 공백이나 대문자 사용 | 소문자 + 하이픈만 사용 |
| push 전에 pull을 안 함 | 항상 `git pull --rebase origin main` 후 push |
| 여러 키워드를 한 커밋에 섞음 | 키워드 하나당 커밋 하나가 원칙 |
| `완료` 상태를 안 바꿈 | 작성 끝나면 반드시 상태 변경 후 커밋 |
